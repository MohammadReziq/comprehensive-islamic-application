================================================================================
  1. السيناريو الكامل: من إعلان الإمام حتى انتهاء المسابقة
================================================================================

تخيل الإمام في مسجد "النور" قرر يعمل مسابقة صلاة:

الخطوة 1: الإمام يفتح التطبيق ← لوحته ← "المسابقات" ← "إنشاء مسابقة"
           يحدد: اسم المسابقة، تاريخ البداية، تاريخ النهاية ← إنشاء.

الخطوة 2: الإمام يعلن في المسجد:
           "في مسابقة صلاة، نزلوا تطبيق صلاتي حياتي، كود المسجد: XXXX"

الخطوة 3: أولياء الأمور ينزلون التطبيق ← يسجلون كـ "ولي أمر"
           (هذا هو الخيار الوحيد المتاح لهم عند التسجيل)
           ← يضيفون أطفالهم ← يربطونهم بالمسجد بالكود.

الخطوة 4: الإمام يفعّل المسابقة (زر "تفعيل").

الخطوة 5: المشرفون يسجلون حضور الأطفال يومياً (QR أو رقم أو اسم).

الخطوة 6: ولي الأمر يرى حضور ابنه يومياً على الرئيسية.

الخطوة 7: المسابقة تنتهي ← الإمام يوقفها ← يرى الترتيب والإحصائيات.


=== سيناريو ولي الأمر ===

أ. سجّل قبل بدء المسابقة:
   - يفتح التطبيق ← يرى: "المسابقة لم تبدأ بعد"
   - أو: "المسابقة ستبدأ في [التاريخ] - باقي X أيام"
   - عندما تبدأ ← البانر يتحول لـ "مسابقة جارية"
   ❌ ناقص: عرض حالة "لم تبدأ" مع التاريخ المتوقع

ب. سجّل بعد بدء المسابقة (متأخر):
   - يسجل ← يضيف ابنه ← يربطه بالمسجد
   - يرى: "مسابقة جارية في مسجد النور"
   - "بدأت: [التاريخ] - تنتهي: [التاريخ] - باقي: X أيام"
   - حضور ابنه يبدأ يتسجل من لحظة الربط (الأيام الفائتة لا تتحسب)
   ❌ ناقص: شاشة تعرض تفاصيل المسابقة + الأيام الفائتة

=== ملاحظة مهمة: الترتيب (Leaderboard) ===
   الترتيب لا يراه إلا الإمام والمشرف فقط.
   ولي الأمر لا يرى ترتيب الأطفال — يرى فقط حضور ابنه ونقاطه.
   السبب: منع المقارنة السلبية بين الأهالي وحماية خصوصية الأطفال.


================================================================================
  2. كيف يعمل الإمام حساب؟ ومن يحدد أنه إمام؟
================================================================================

النظام الحالي:
- عند التسجيل: خياران فقط = "ولي أمر" أو "إمام"
  (لا يظهر "مشرف" ولا "سوبر أدمن" — انظر القسم 3 لشرح المشرف)
- إذا اختار "إمام" ← دوره يكون imam في قاعدة البيانات
- يدخل ← بوابة المسجد ← "إنشاء مسجد جديد"
- يملأ البيانات ← يرسل طلب
- الطلب يروح للسوبر أدمن
- السوبر أدمن يوافق أو يرفض
- إذا وافق ← الإمام يرى لوحته الكاملة

يعني: لا أحد "يشير" مين بده يكون إمام!
الإمام ينشئ حسابه بنفسه ويختار "إمام".
لكن ما يشتغل حتى السوبر أدمن يوافق على مسجده.
هذا هو "التحقق" — السوبر أدمن هو اللي بيأكد.

السوبر أدمن:
- حساب خاص خارج التسجيل العادي
- يُنشأ يدوياً في قاعدة البيانات (مرة واحدة)
- دوره فقط: الموافقة/الرفض على طلبات المساجد
✅ موجود ويعمل


================================================================================
  3. كيف يضيف الإمام المشرفين؟ (تصميم جديد مقترح)
================================================================================

=== التصميم الحالي (المشكلة) ===
حالياً عند التسجيل يظهر خيار "إمام / مشرف" — أي شخص يقدر يختار
"مشرف" حتى لو ما حد بده يكون مشرف. هذا يسبب:
- ناس تتزانخ تختار "مشرف" وهي مش مشرفة
- ارتباك في الأدوار
- حسابات مشرف بدون مسجد

=== التصميم الجديد المقترح ===

المبدأ: عند التسجيل يظهر خياران فقط: "ولي أمر" أو "إمام"
        المشرف لا يسجل بنفسه — الإمام هو اللي ينشئ حساب المشرف.

الطريقة:
1. الإمام يفتح لوحته ← "إدارة المشرفين" ← "إضافة مشرف جديد"
2. يدخل: اسم المشرف + رقم الهاتف أو الإيميل + كلمة سر مؤقتة
3. النظام ينشئ حساب للمشرف تلقائياً (بدور supervisor)
4. الإمام يعطي المشرف بيانات الدخول (شفهياً أو واتساب):
   "إيميلك: xxx@xxx.com — كلمة السر: 12345 — غيرها عند أول دخول"
5. المشرف يفتح التطبيق ← يسجل دخول ← يدخل مباشرة للوحة المشرف

المزايا:
+ لا ارتباك في الأدوار — ولي الأمر يسجل كولي أمر وبس
+ الإمام يتحكم بمن يكون مشرف
+ لا حسابات مشرف يتيمة بدون مسجد
+ المشرف يبدأ يشتغل فوراً (ما يحتاج يبحث عن كود دعوة)

ما يحتاج تنفيذ:
- Edge Function أو RPC: create_supervisor_account(name, email, password, mosqueId)
  تنشئ auth user + users row (role=supervisor) + mosque_members row
- شاشة في لوحة الإمام: "إضافة مشرف" (فورم بسيط)
- عرض بيانات الدخول المؤقتة للإمام (مرة واحدة فقط)
- إزالة خيار "إمام / مشرف" من التسجيل ← يصير "ولي أمر" أو "إمام" فقط

ماذا يحصل للنظام القديم (كود الدعوة)؟
- يبقى كود الدعوة كبديل احتياطي إذا الإمام ما يبي ينشئ حسابات
- أو يُحذف بالكامل لتبسيط التجربة (القرار لك)

ملاحظة: هذا تغيير في التصميم يحتاج:
❌ migration جديدة (إذا أضفنا علاقة مباشرة)
❌ Edge Function (لإنشاء حساب auth + ربط بالمسجد بعملية واحدة)
❌ تعديل شاشة التسجيل (حذف خيار المشرف)
❌ شاشة إضافة مشرف في لوحة الإمام


================================================================================
  4. تدفق ولي الأمر (تسجيل مبكر vs متأخر)
================================================================================

أ. سجّل قبل إطلاق المسابقة:
   1. يسجل كـ "ولي أمر" ← يضيف أطفاله ← يربطهم بالمسجد
   2. يفتح التطبيق ← يرى: "المسابقة لم تبدأ بعد"
      أو: "المسابقة ستبدأ في [التاريخ] — باقي X أيام"
   3. عندما الإمام يفعّل المسابقة ← البانر يتحول لـ "مسابقة جارية"
   ❌ ناقص: عرض حالة "لم تبدأ" بشكل واضح

ب. سجّل بعد بدء المسابقة:
   1. يسجل ← يضيف ابنه ← يربطه بالمسجد
   2. يرى: "مسابقة جارية في مسجد [الاسم]"
      + تاريخ البداية والنهاية + الأيام المتبقية
   3. حضور ابنه يبدأ من لحظة الربط
   ❌ ناقص: شاشة تفاصيل المسابقة


================================================================================
  5. فصل الأدوار الكامل
================================================================================

سوبر أدمن:
  يرى: طلبات المساجد فقط (موافقة/رفض)
  لا يرى: أطفال، حضور، مسابقات، أي شيء آخر
  ✅ مكتمل

الإمام (مالك المسجد):
  يرى:
  - معلومات المسجد + الأكواد (كود المسجد + كود الدعوة)
  - المشرفون (قائمة + إضافة + حذف)
  - طلبات الانضمام (قبول/رفض)
  - الطلاب (قائمة + تفاصيل كل طالب)
  - التحضير (مسح QR أو رقم محلي أو اسم)
  - حضور اليوم
  - طلبات التصحيح (مراجعة + موافقة/رفض)
  - الملاحظات (إرسال ملاحظة لولي الأمر عن طفل)
  - المسابقات (إنشاء + تفعيل + إيقاف + الترتيب + الإحصائيات)
  - الإعلانات (إرسال)
  ✅ معظمه مكتمل
  ❌ ناقص: إحصائيات ما بعد المسابقة، تعديل/حذف المسابقة، تحديد موقع المسجد
  ❌ ناقص: شاشة إنشاء حساب مشرف (التصميم الجديد)

المشرف:
  يرى: نفس لوحة الإمام لكن بدون:
  - إنشاء مسابقات (يرى الترتيب فقط)
  - إدارة المشرفين
  - أكواد الدعوة
  - إعدادات المسجد
  يقدر: تحضير، طلاب، تصحيحات، ملاحظات، عرض الترتيب
  ✅ معظمه مكتمل
  ❌ ناقص: بانر حالة المسابقة على لوحته

ولي الأمر:
  يرى:
  - حضور أطفاله اليوم (مع Realtime)
  - أطفالي (إضافة، بطاقة QR، ربط بمسجد)
  - بانر المسابقة (جارية/قادمة/منتهية) — بدون ترتيب
  - ملاحظات المشرف
  - الإعلانات
  لا يرى: ترتيب الأطفال (الترتيب خاص بالإمام والمشرف فقط)
  ✅ معظمه مكتمل
  ❌ ناقص: طلب تصحيح من بطاقة الطفل


================================================================================
  6. تفاصيل ما بعد المسابقة للإمام
================================================================================

عندما تنتهي المسابقة، الإمام لازم يشوف:

أ. الترتيب النهائي:
   - قائمة الأطفال مرتبة بالنقاط (الأول، الثاني، الثالث...)
   ✅ موجود (getLeaderboard في CompetitionRepository)

ب. إحصائيات المسابقة الشاملة:
   - عدد المشاركين الإجمالي
   - عدد أيام المسابقة
   - إجمالي سجلات الحضور
   - نسبة الالتزام العامة (حضور فعلي / حضور متوقع × 100)
   - أفضل يوم حضور / أسوأ يوم
   - متوسط الحضور اليومي
   ❌ ناقص (يحتاج RPC في Supabase)

ج. تقرير لكل طفل:
   - كم صلاة حضر من أصل كم
   - نسبة الالتزام
   - أطول سلسلة حضور خلال المسابقة
   - أيام الغياب
   ❌ ناقص (يحتاج شاشة تقرير مفصل)

د. تصدير النتائج:
   - تصدير كصورة أو PDF (للطباعة أو المشاركة في مجموعة واتساب)
   ❌ ناقص (ميزة إضافية)


================================================================================
  7. التكامل بين الأدوار (Real-time)
================================================================================

المبدأ: فعل من دور = تأثير فوري على الأدوار الأخرى.

الإمام يفعّل المسابقة ←
  المشرف: يرى "مسابقة جارية" على لوحته          ❌ ناقص
  ولي الأمر: يرى بانر "مسابقة جارية"             ❌ ناقص

الإمام يوقف المسابقة ←
  المشرف: يرى "المسابقة انتهت"                   ❌ ناقص
  ولي الأمر: يرى "المسابقة انتهت"                 ❌ ناقص

المشرف يسجل حضور ←
  ولي الأمر: يرى فوراً "ابنك حضر [الصلاة]"       ✅ موجود (Realtime)

ولي الأمر يطلب تصحيح ←
  الإمام/المشرف: يرى "طلب تصحيح جديد"             ❌ لا Realtime

المشرف يرسل ملاحظة ←
  ولي الأمر: يرى ملاحظة جديدة                     ❌ لا Realtime

السوبر أدمن يوافق على مسجد ←
  الإمام: ينتقل للوحته تلقائياً                    ✅ موجود (Realtime)

ولي الأمر يربط ابنه بمسجد ←
  المشرف: يرى الطالب الجديد فوراً                  ✅ موجود (Realtime)

ملخص Realtime:
  ✅ attendance, mosques, mosque_children
  ❌ correction_requests, notes, competitions (حالة), announcements


================================================================================
  8. نظام الأرقام المحلية (local_number) وتعارض المساجد
================================================================================

=== كيف يعمل النظام الحالي ===
كل طفل عندما يرتبط بمسجد يأخذ رقم محلي (local_number).
هذا الرقم فريد داخل المسجد الواحد فقط (مثلاً: 1, 2, 3...).

مثال:
  مسجد النور: أحمد = رقم 1، خالد = رقم 2، سعد = رقم 3
  مسجد الفجر: يوسف = رقم 1، عمر = رقم 2

يعني: الرقم 1 موجود في مسجد النور وفي مسجد الفجر — لا تعارض!
لأن الأرقام مفصولة بالمسجد (mosque_id + local_number).

=== ماذا يحصل لو الطفل ارتبط بمسجد فرعي؟ ===
إذا أحمد (رقم 1 في مسجد النور) ارتبط بمسجد الفجر كمسجد ثانوي:
  - يأخذ رقم جديد في مسجد الفجر (مثلاً رقم 3)
  - ما في تصادم لأن كل مسجد عنده أرقام مستقلة

=== هل في مشكلة Race Condition؟ ===
نعم، ممكن: إذا وليان أمر ربطا ابنيهما بنفس المسجد بنفس اللحظة.
الحل الموجود حالياً:
  - الكود يحسب MAX(local_number) + 1
  - إذا حصل تعارض (unique violation) ← يعيد الحساب ويحاول مرة ثانية
  - حتى 3 محاولات (retry loop)
  ✅ هذا موجود ويعمل (child_repository.dart سطر 140-167)

=== تحسين مقترح ===
بدل retry في التطبيق، الأفضل:
  - RPC في Supabase: SELECT MAX(local_number) + 1 ... FOR UPDATE
  - يضمن عدم التعارض بعملية قاعدة بيانات واحدة
  - أنظف وأضمن من retry

  9. نقد ذاتي للدراسة السابقة + تصحيحات
================================================================================

=== أخطاء في النسخة 1.0 ===

خطأ 1: "الترتيب لولي الأمر"
  ✗ كتبت أن ولي الأمر يرى ترتيب أطفاله في المسابقة
  ✓ التصحيح: الترتيب (Leaderboard) يراه الإمام والمشرف فقط.
  ولي الأمر يرى: حضور ابنه + نقاطه فقط — بدون مقارنة بأطفال آخرين.

خطأ 2: "AttendanceValidationService غير مبني"
  ✗ كتبت أنه غير مبني
  ✓ التصحيح: الخدمة مبنية بالكامل وتعمل!
  attendance_validation_service.dart موجود ومربوط في recordAttendance.
  يتحقق من: وقت الأذان، نافذة التسجيل، إحداثيات المسجد.

خطأ 3: "الإمام ينشئ حسابه ويختار إمام/مشرف"
  ✗ خلطت بين الإمام والمشرف في خيارات التسجيل
  ✓ التصحيح: التسجيل حالياً يعرض "ولي أمر" أو "إمام / مشرف".
  لكن حسب التصميم الجديد (القسم 3): يجب إزالة المشرف من التسجيل.

خطأ 4: لم أذكر نظام الأرقام المحلية
  ✗ ما ذكرت كيف يتعامل النظام مع تعارض الأرقام
  ✓ التصحيح: أضفت قسم 8 كامل يشرح النظام + حالة المسجد الفرعي.

=== نقاط ضعف في النسخة 1.0 ===

ضعف 1: لم أميز بمن يرى ماذا بدقة
  مثلاً كتبت "شاشة المسابقة النشطة لولي الأمر" وأعطيتها ترتيب
  بينما الأصح: ولي الأمر يرى حالة المسابقة (جارية/لا) + حضور ابنه فقط.

ضعف 2: لم أناقش تغيير تدفق المشرف
  الدراسة السابقة افترضت أن المشرف يسجل بنفسه — لكن هذا يسبب مشاكل.

ضعف 3: لم أناقش حالات الحافة (Edge Cases)
  مثلاً: ماذا لو الإمام حذف مشرف؟ ماذا لو ولي الأمر فك ربط ابنه أثناء المسابقة؟

ضعف 4: لم أذكر إلغاء الحضور
  الإمام والمشرف يقدرون يلغون حضور خاطئ — هذا موجود بالفعل!
  cancelAttendance RPC مبني ويعمل.

✅ المسابقات (عرض الترتيب فقط)

=== ولي الأمر ===
✅ الرئيسية (حضور اليوم + Realtime)
✅ أطفالي (إضافة، بطاقة QR، ربط بمسجد)
✅ الملاحظات + الرد عليها
✅ الإعلانات + بانر المسابقة (جزئي)

=== السوبر أدمن ===
✅ طلبات المساجد — مكتمل بالكامل


================================================================================
  11. ما هو ناقص ولازم يتبنى (❌) — مُحدَّث
================================================================================

الأولوية العالية (قبل الإطلاق):

❌ 1. تغيير شاشة التسجيل
      حذف خيار "مشرف" من التسجيل
      يبقى: "ولي أمر" أو "إمام" فقط
      المدة: نصف يوم

❌ 2. الإمام ينشئ حسابات المشرفين
      Edge Function: create_supervisor_account
      شاشة في لوحة الإمام: "إضافة مشرف"
      عرض بيانات الدخول المؤقتة
      المدة: يومين

❌ 3. شاشة طلب التصحيح لولي الأمر
      الريبو والبلوك جاهزان — ناقص الواجهة فقط
      زر من بطاقة الطفل ← شاشة (صلاة + تاريخ + ملاحظة + مسجد) ← إرسال
      المدة: يوم واحد

❌ 4. موقع المسجد (الإمام يحدد lat/lng)
      شاشة إعدادات ← خريطة أو GPS ← حفظ
      ربط أوقات الصلاة بالإحداثيات
      المدة: يومين

❌ 5. بانر المسابقة بجميع الحالات
      "لا مسابقة" / "ستبدأ [التاريخ]" / "جارية في مسجد [الاسم]" / "انتهت"
      يظهر على: لوحة الإمام + المشرف + رئيسية ولي الأمر
      المدة: يوم واحد


الأولوية المتوسطة:

❌ 6. إحصائيات ما بعد المسابقة (الإمام)
      عدد المشاركين، نسبة الالتزام، أفضل/أسوأ يوم، تقرير لكل طفل
      المدة: يومين

❌ 7. تعديل وحذف المسابقة
      تعديل الاسم/التواريخ + حذف (مع تحقق)
      المدة: يوم واحد

❌ 8. Realtime لطلبات التصحيح والملاحظات والمسابقات
      subscribe في RealtimeService
      المدة: يوم واحد

❌ 9. تحديث البيانات عند العودة للشاشة (RouteAware)
      المدة: نصف يوم

❌ 10. ربط جهاز الابن (عرض الباركود من جوال الابن)
       كود ربط مؤقت + شاشة ابن محدودة
       المدة: 3 أيام


الأولوية المنخفضة:

❌ 11. المشرف في أكثر من مسجد (شاشة اختيار)
❌ 12. الإعلانات للمشرفين (عمود target_audience)
❌ 13. أوقات الصلاة حسب موقع المسجد
❌ 14. Offline Mode
❌ 15. مسجد أساسي + ثانوي (واجهة)
❌ 16. تصدير نتائج المسابقة

  1.1 تعديل شاشة التسجيل (حذف خيار المشرف)     — نصف يوم
  1.2 Edge Function + شاشة إنشاء حساب مشرف       — يومين
