القسم الأول: الأدوار وصلاحياتها
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

┌──────────────────────────────────────────────────┐
│ 1. الإمام (المدير المطلق)                        │
├──────────────────────────────────────────────────┤
│ • تسجيل حضور لأي صلاة ماضية، لأي يوم، بلا قيود.│
│   (المستقبل محظور حتى على الإمام — قاعدة صلبة). │
│ • تعديل أي سجل قديم بأثر رجعي بلا حدود زمنية.  │
│ • اعتماد ورفض طلبات التصحيح.                    │
│ • مراقبة سجل نشاط (Audit Log) لجميع المشرفين.   │
│ • التحكم بإعدادات صلاحيات المشرفين.              │
│ • تعليق صلاحيات مشرف أو إزالته.                 │
│ وهو المرجعية النهائية لأي خلاف.                 │
└──────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────┐
│ 2. المشرف                                        │
├──────────────────────────────────────────────────┤
│ • تسجيل/تعديل الحضور ← آخر 3 أيام فقط.         │
│ • اعتماد/رفض طلبات التصحيح المقدمة من الأهالي.  │
│ • الصلوات المستقبلية ← محظورة تماماً.            │
│ • الصلوات أقدم من 3 أيام ← للقراءة فقط.         │
│ ملاحظة: المشرف الجديد لا يرى طلبات سابقة        │
│ لتاريخ انضمامه (لأنه لم يشهد تلك الفترة).       │
└──────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────┐
│ 3. ولي الأمر                                     │
├──────────────────────────────────────────────────┤
│ • مشاهدة سجل حضور أبنائه (للقراءة فقط).         │
│   مع ذكر: من سجّل الحضور/الغياب + متى.          │
│ • إرسال "طلب تصحيح" في أي وقت لأي صلاة قديمة.  │
│ • متابعة حالة الطلب (معلق / مقبول / مرفوض).     │
│ • استلام إشعارات داخلية عند الرد على طلبه.       │
│ • حد أقصى: 3 طلبات معلقة في نفس الوقت.          │
│   (لمنع إغراق المشرفين بطلبات متكررة).          │
│ لا يملك أي صلاحية لتعديل السجلات مباشرة.        │
└──────────────────────────────────────────────────┘

القسم الثاني: النظرية الزمنية الثلاثية (3-Tier Time Model)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

⚠ مبدأ أساسي: كل الفحوصات الزمنية تتم على السيرفر (Server-Side
Timestamp) وليس على ساعة الهاتف. هذا يمنع التلاعب بالتوقيت المحلي.
الواجهة (Client) تعرض القيود بصرياً، لكن السيرفر هو الحَكَم النهائي.

╔══════════════════════════════════════════════════╗
║ المرحلة 1: المستقبل (مَنع مطلق على الجميع)      ║
╠══════════════════════════════════════════════════╣
║ لا يمكن لأي أحد (حتى الإمام) تسجيل صلاة لم     ║
║ يدخل وقتها. القائمة لا تُظهر صلاة لم يؤذَّن لها.║
║ هذا ثابت برمجياً ومنطقياً وشرعياً.               ║
║ الحماية: Client-Side (لا يظهر الخيار أصلاً)      ║
║        + Server-Side (يرفض الطلب حتى لو أُرسل).  ║
╠══════════════════════════════════════════════════╣
║ شبهة: "لماذا لا نحضّر القائمة مسبقاً؟"          ║
║ الرد: يفتح باب التسجيل المسبق بالخطأ.            ║
║ المشرف قد ينسى القائمة "حاضرة" لمن غاب.         ║
║ يضر بمصداقية المنافسات (Leaderboard).            ║
╚══════════════════════════════════════════════════╝

╔══════════════════════════════════════════════════╗
║ المرحلة 2: نافذة الـ 72 ساعة (مرونة كاملة)      ║
╠══════════════════════════════════════════════════╣
║ المشرف يسجل/يعدل الحضور بحرية لآخر 3 أيام.     ║
║ لا يحتاج طلبات رسمية، يدخل ويعدل مباشرة.       ║
║ ولي الأمر يستطيع أيضاً إرسال طلب تصحيح خلال    ║
║ هذه الفترة، والمشرف يعالجه مباشرة أو عبر الطلب. ║
╠══════════════════════════════════════════════════╣
║ لماذا 3 أيام بالتحديد؟                          ║
║ • يوم واحد: غير واقعي (المشرف متطوع قد ينسى).   ║
║ • أسبوع: مبالغ (الذاكرة لا تتذكر بعد 6 أيام).  ║
║ • 3 أيام: التوازن المثالي. قابل للتعديل من       ║
║   الإمام في الإعدادات (1 أو 3 أو 7 أيام).       ║
╚══════════════════════════════════════════════════╝

╔══════════════════════════════════════════════════╗
║ المرحلة 3: الإغلاق (ما بعد 72 ساعة)             ║
╠══════════════════════════════════════════════════╣
║ المشرف: شاشة للقراءة فقط (Read-Only).           ║
║ التعديل: فقط عبر "طلب تصحيح" من ولي الأمر.     ║
║ الاستثناء الوحيد: الإمام (بلا حدود زمنية).      ║
╚══════════════════════════════════════════════════╝

القسم الثالث: دورة حياة "طلب التصحيح" الكاملة
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

--- الخطوة 1: ولي الأمر يرسل الطلب ---
• يفتح بطاقة ابنه ← يضغط "طلب تصحيح حضور".
• يختار: المسجد (فقط المربوط بالطفل)، الصلاة، التاريخ، ملاحظة.
• حد أقصى 3 طلبات معلقة بنفس الوقت (Anti-Spam).
• يُرسل الطلب ← حالته "معلق".

--- الخطوة 2: الطلب يظهر عند المشرفين (Shared Inbox) ---
• يظهر لجميع مشرفي المسجد في شاشة "الطلبات المعلقة".
• ليس لمشرف بعينه! بل لكل المشرفين المسجلين بالمسجد.
• المشرفون الذين انضموا بعد تاريخ الطلب لا يرونه.
• أول مشرف يفتح الطلب ويتعامل معه، يُغلق للباقين تلقائياً.
• لو تجاهل الجميع: بعد 7 أيام يُصعّد تلقائياً للإمام مع
  إشعار: "هناك طلبات تصحيح معلقة لم تُعالج".

--- الخطوة 3: المشرف يتخذ القرار ---
خيار "قبول":
  • يضغط قبول ← يتعدل سجل الحضور تلقائياً.
  • ينتقل الطلب لحالة "مقبول".
  • يصل إشعار داخلي لولي الأمر: "تم قبول طلبك".

خيار "رفض":
  • يضغط رفض ← تظهر قائمة إجبارية لسبب الرفض:
    - "الابن لم يتواجد في المسجد"
    - "صلى في مسجد آخر"
    - "سبب آخر" (مع حقل نص حر)
  • يصل إشعار لولي الأمر: "تم رفض طلبك بسبب: ..."
  • فائدة الأسباب الإجبارية: تحمي المشرف من الإحراج
    الشخصي وتحوّل الخلاف لنقاش تقني محايد.

--- الخطوة 4: الإمام يشرف ---
• يرى جميع الطلبات (المعلقة + المقبولة + المرفوضة).
• يمكنه التدخل: إلغاء قبول خاطئ، أو قبول طلب مرفوض.
• يرى الـ Audit Log: أي مشرف قبل كم طلب، وكم رفض.
• يمكنه تعليق صلاحيات مشرف لو لاحظ أنماطاً مشبوهة.
• يستلم إشعار تلقائي لو طلب ظل معلقاً 7 أيام (Escalation).

--- حالة خاصة: الأب أرسل طلباً والصلاة ضمن 3 أيام ---
• المشرف يستطيع تعديل الحضور مباشرة من شاشة التسجيل.
• النظام يكتشف تلقائياً أن الطلب أصبح محلولاً ← يغلقه.
• ولا يحتاج المشرف لفتح شاشة الطلبات أصلاً.

القسم الرابع: تصميم واجهة "تسجيل الحضور" (منع العجقة)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

المشكلة: المشرف يفتح التطبيق وقت العصر، يريد تسجيل الفجر.
كيف نمنعه من الخطأ (تسجيل الفجر في خانة العصر)؟

1. الوضع الافتراضي عند فتح الشاشة (Smart Default):
   • التطبيق يسأل السيرفر عن "الصلاة الحالية" (لا يعتمد
     على ساعة الهاتف). يفتح على تلك الصلاة مع تاريخ اليوم.
   • مثال: فتح الساعة 2 ظهراً ← "صلاة الظهر - اليوم".

2. تغيير الصلاة (اختياري وواعي):
   • زر بارز في الأعلى: "صلاة الظهر - اليوم ▼"
   • بالضغط عليه تظهر نافذة اختيار بسيطة:
     الصلاة: [الفجر] [الظهر] [العصر] ... (فقط ما حان وقته)
     التاريخ: [اليوم] [أمس] [أول أمس]
   • لا تظهر أي صلاة مستقبلية ولا تاريخ أقدم من 3 أيام.

3. الحماية البصرية (Color Coding + أيقونات + نص):
   • الفجر: خلفية زرقاء + أيقونة هلال + نص "الفجر"
   • الظهر: خلفية صفراء + أيقونة شمس + نص "الظهر"
   • العصر: خلفية برتقالية + أيقونة غروب + نص "العصر"
   • المغرب: خلفية حمراء + أيقونة أفق + نص "المغرب"
   • العشاء: خلفية كحلية + أيقونة نجوم + نص "العشاء"
   ⚠ ملاحظة لعمى الألوان: لا نعتمد على اللون وحده أبداً.
   كل صلاة لها أيقونة مختلفة ونص واضح بالإضافة للون.

4. شريط تنبيه ذكي (عند التسجيل بأثر رجعي):
   • إذا اختار صلاة غير الحالية أو تاريخاً سابقاً:
   • يظهر شريط: "⚠ أنت تسجل حضور صلاة الفجر ليوم أمس"

5. شاشة مراجعة قبل الحفظ (Confirmation Screen):
   • بعد تحديد جميع الطلاب (حاضر/غائب)، يضغط "حفظ".
   • تظهر شاشة ملخص: "ستسجل 25 حاضر و 3 غائب لصلاة
     الفجر - اليوم. هل أنت متأكد؟"
   • زرّان: [تأكيد] [مراجعة]
   • هذا يمنع الأخطاء الجماعية (ضغط حاضر للكل بالخطأ).

القسم الخامس: نظام الإشعارات الداخلية (In-App Notifications)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

لا نستخدم Push Notifications (Firebase/APNs) في هذه المرحلة.
بدلاً من ذلك، نبني نظام إشعارات داخلي بسيط وقوي:

الآلية:
• جدول "notifications" في Supabase.
• أيقونة جرس (🔔) في الشريط العلوي لجميع الأدوار.
• عدّاد أحمر (Badge) يظهر عدد الإشعارات غير المقروءة.
• صفحة إشعارات كاملة مرتبة زمنياً.
• تنظيف تلقائي: حذف الإشعارات المقروءة أقدم من 30 يوماً.
  (أو أرشفتها في جدول notifications_archive).

أنواع الإشعارات:

لولي الأمر:
• "تم قبول طلب تصحيح الحضور لابنك أحمد - صلاة الفجر"
• "تم رفض طلبك. السبب: الابن لم يكن في المسجد"
• "ملاحظة جديدة من المشرف بخصوص ابنك"
• "إعلان جديد من مسجد الرحمة"

للمشرف:
• "طلب تصحيح جديد من ولي أمر أحمد"
• "الإمام ألغى قبولك لطلب تصحيح رقم #12"
• "إعلان جديد من الإمام"

للإمام:
• "طلب انضمام مشرف جديد للمسجد"
• "تنبيه: 3 طلبات تصحيح معلقة منذ 7 أيام" (Escalation)
• "ملخص أسبوعي: المشرف محمد قبل 5 طلبات، رفض 2"

القسم السادس: إعدادات الإمام (لوحة التحكم)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. مدة التسجيل المباشر للمشرفين:
   [1 يوم] [3 أيام ✓ افتراضي] [7 أيام]

2. السماح للمشرفين بقبول طلبات التصحيح:
   [مُفعّل ✓ افتراضي] / [معطّل - فقط الإمام يقبل]

3. إدارة المشرفين:
   • رؤية قائمة المشرفين + تاريخ انضمام كل واحد.
   • تعليق صلاحيات مشرف (مع إبقائه عضواً).
   • إزالة مشرف من المسجد.

القسم السابع: جميع السيناريوهات المتطرفة (Edge Cases)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

السيناريو 1: جميع المشرفين غابوا لأكثر من 3 أيام
→ الإمام يملك Override مطلق. يدخل ويسجل نيابة عنهم.
→ بعد 7 أيام من طلب معلق = Escalation تلقائي للإمام.

السيناريو 2: الأب لاحظ خطأً بعد ساعتين
→ يرسل طلب تصحيح فوراً (حتى لو ضمن الـ 3 أيام).
→ المشرف يرى الطلب أو يعدل مباشرة.
→ النظام يغلق الطلب تلقائياً إذا تم الإصلاح المباشر.

السيناريو 3: أب كاذب يرسل طلبات متكررة
→ حد أقصى 3 طلبات معلقة بنفس الوقت (Anti-Spam).
→ المشرف يرفض بسبب إجباري ← يصل للأب.
→ الإمام يرى الأنماط المتكررة في الـ Audit Log.

السيناريو 4: خطأ لمسي (Fat Finger)
→ Color Coding + أيقونات + نص = حماية ثلاثية.
→ Smart Default يفتح على الصلاة الحالية.
→ شاشة تأكيد قبل الحفظ (Confirmation Screen).

السيناريو 5: مشرف تلاعب بالنقاط لصالح طالب
→ Audit Log يسجل كل تعديل باسم المشرف والتاريخ.
→ الإمام يراقب ← يعلّق صلاحياته لو اكتشف تلاعب.

السيناريو 6: طفل مسجل في مسجدين
→ كل مسجد له سجل حضور مستقل.
→ "حاضر" في مسجد أ لا يعني "حاضر" في مسجد ب.
→ ولي الأمر يختار المسجد عند إرسال طلب التصحيح.

السيناريو 7: مشرفان ضغطا "قبول" لنفس الطلب بنفس اللحظة
→ Database-Level Lock (Optimistic Locking).
→ أول ضغطة تُنفذ. الثاني يرى: "تم معالجة الطلب مسبقاً".

السيناريو 8: المشرف غيّر ساعة هاتفه للتلاعب بالأيام
→ كل الفحوصات على السيرفر (Server Timestamp).
→ حتى لو الهاتف يقول "أمس"، السيرفر يعرف الحقيقة.
→ الواجهة للعرض فقط، والسيرفر هو الحَكَم.

السيناريو 9: مشرف جديد انضم بعد الحدث
→ لا يرى طلبات تصحيح سابقة لتاريخ انضمامه.
→ يرى فقط الطلبات الجديدة القادمة بعد انضمامه.

السيناريو 10: طالب حاضر لكن المشرف لم يره (صف خلفي)
→ ولي الأمر يرسل طلب تصحيح مع ملاحظة: "كان في الصف الخلفي".
→ المشرف يقرر بناءً على معرفته. لو شك ← يسأل الإمام.
→ النظام لا يحكم على الواقع، بل يوفر آلية تصحيح عادلة.

السيناريو 11: مشرف سجّل طالباً غائباً "حاضر" بالخطأ
→ اكتشاف الخطأ: ضمن 3 أيام يعدل المشرف مباشرة.
→ بعد 3 أيام: الإمام فقط يستطيع التعديل.
→ مستقبلاً: تقرير "أنماط غير اعتيادية" (مثلاً طالب
  حاضر 100% ← تنبيه بسيط للمراجعة).

السيناريو 12: رمضان والصلوات الإضافية (تراويح/قيام)
→ المرحلة الأولى: الصلوات الخمس فقط.
→ مستقبلاً: إضافة صلوات مخصصة (تراويح، جمعة...) كخيار
  يفعّله الإمام من إعدادات المسجد.

القسم الثامن: الحماية التقنية (Defense in Depth)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

كل عملية كتابة في قاعدة البيانات محمية بطبقتين:

الطبقة 1 (الواجهة - Client):
• لا يظهر زر أو خيار لعملية غير مسموحة.
• مثال: المشرف لا يرى تواريخ أقدم من 3 أيام أصلاً.

الطبقة 2 (السيرفر - Supabase RLS + Database Functions):
• Row-Level Security Policies تمنع أي تعديل غير مصرح.
• Database Functions تفحص الأوقات بـ Server Timestamp.
• حتى لو استخدم شخص API مباشرة، السيرفر يرفض.

Audit Log:
• جدول attendance_audit يسجل: من عدّل، ماذا عدّل، متى.
• لا يمكن حذفه أو تعديله (Append-Only).
• الإمام يراه من لوحة القيادة في أي وقت.

Concurrency (التزامن):
• طلبات التصحيح: Optimistic Locking (version column).
• تسجيل الحضور: UPSERT (لمنع التكرار).
• لا يمكن تسجيل نفس الطالب لنفس الصلاة مرتين.

Database Maintenance:
• الإشعارات المقروءة > 30 يوم ← أرشفة أو حذف تلقائي.
• Indexes على: mosque_id, prayer_date, child_id.
• Pagination في كل الشاشات (20 عنصر/صفحة).

القسم التاسع: الخلاصة القاطعة
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

هذا النظام يحقق التوازن الكامل بين 4 محاور:

1. المرونة: المشرف لا يُحبط (3 أيام كافية، قابلة للتعديل).
2. الموثوقية: ولي الأمر لديه قناة رسمية دائمة.
3. الكفاءة: الإمام لا يصبح عنق زجاجة (مع Escalation تلقائي).
4. الأمان: أخطاء بشرية ممنوعة بالتصميم (Smart UI + Server).

الدورة الكاملة:
أذان ← مشرف يسجل ← 3 أيام مرنة ← إغلاق ← طلب تصحيح
← Shared Inbox ← مشرف يوافق/يرفض ← إشعار داخلي للأب
← إمام يراقب بـ Audit Log ← Escalation بعد 7 أيام

النظام متكامل الحلقات، يوازن بين "الثقة التطوعية" المطلوبة
للعمل الدعوي في المسجد، و"الانضباط التقني" (Data Integrity)
المطلوب للمنافسة العادلة، مع حماية مزدوجة (Client + Server)
وقابلية توسع مستقبلية (حلقات، صلوات إضافية، Push Notifications).
